## 为什么要在聊天应用里模拟天体物理？

Echo 是一款强调"极客美学、强隐私、高纯净度"的云原生即时通讯应用。它的视觉系统不满足于扁平化的 Material Design，而是试图回答一个问题：

**两个用户连续聊天 365 天后，界面应该长什么样？**

答案是——像两颗互相环绕的恒星。

这套名为"星轨共鸣 (Star Track Resonance)"的视觉系统，根据通信双方的**连续通讯天数**（基于 `Pair<UserIdA, UserIdB>` 严格隔离），分 8 个层级渐进演化：

| 天数 | 境界 | 视觉效果 |
|------|------|---------|
| 3-6 天 | 星尘 | Canvas 粒子背景，像素级白点缓慢漂浮 |
| 7-13 天 | 引力线 | SVG 连线渲染在不同发送者消息卡片之间 |
| 14-29 天 | 潮汐锁定 | CSS 3D perspective 微旋转，己方 -1deg 对方 +1deg |
| 30-49 天 | 双子星 | 背景双色渐变交织，像两颗星互相盘旋 |
| 50-98 天 | 视界线 | 接近视口边缘的消息卡片触发拉伸模糊 |
| 99-179 天 | 超新星 | 包含情感词汇时触发 0.5s 局部错位动画 |
| 180-364 天 | 脉冲星 | 顶部 2px 光点按 60 BPM 闪烁 |
| 365+ 天 | 微波背景辐射 | 滑到顶部渐显 SVG 彩色噪点图 |

## 状态隔离：为什么 Pair 粒度是唯一解？

Echo 的视觉系统有一个硬性约束：

> **所有关于"连续通讯天数"的状态，必须基于 `Pair(UserIdA, UserIdB)` 进行严格隔离。**

这条约束的根源来自云原生架构的核心理念——**无状态容器**。

### 后端：无状态容器的天然约束

在 Docker Compose 三容器架构下，Echo 的后端服务本质上是无状态的。任何需要持久化的"连续天数"计算都必须下沉到 PostgreSQL，而不是缓存在 Node.js 进程内存中。原因是：

```
容器重启 → 内存丢失 → 所有 Pair 状态归零
```

因此后端的连续天数计算完全基于数据库查询：

```sql
-- 伪代码：按 Pair 聚合消息日期
SELECT DATE(created_at) as msg_date
FROM messages
WHERE (sender_id, receiver_id) IN ((:a, :b), (:b, :a))
GROUP BY DATE(created_at)
ORDER BY msg_date
```

然后由应用层计算连续天数（按日期排序，检查相邻日期间隔），而非简单的 `COUNT(DISTINCT DATE)`。

### 前端：Pair Key 的确定性缓存

前端使用有序 Pair Key 作为缓存键：

```typescript
function pairKey(a: string, b: string): string {
  return [a, b].sort().join('::');
  // 无论谁发起请求，UserA::UserB 的键始终相同
}
```

这个设计确保：
1. **缓存可预测**：同一对用户在不同会话中的缓存键一致
2. **无竞态**：不依赖消息方向（谁发给谁）
3. **可驱逐**：LRU 策略按 Pair 维度淘汰，不影响其他 Pair

### 与云原生的呼应

这种 Pair 级别的状态隔离与 Kubernetes 的 Pod 设计哲学异曲同工：

- **Pod ↔ Pair**：最小调度单元，状态自包含
- **ConfigMap ↔ Pair Key**：确定性标识，跨实例一致
- **Volume ↔ PostgreSQL**：持久化存储，进程重启不丢失

当后端的每个 API 请求都携带完整的认证上下文（JWT → userId），任何负载均衡器都能把请求路由到任意容器实例——这正是无状态架构的核心优势。

## 性能瓶颈与优化：物理 UI 不是免费的

### 问题 1：Canvas 粒子动画的 CPU 预算

3 天以上的聊天窗口会挂载一个全屏 Canvas 渲染 40 个像素级粒子。第一版使用 `requestAnimationFrame` 持续绘制，在低端 Android WebView 上帧率跌到 30fps。

**优化策略**：
- 粒子数量动态缩放：`Math.min(40, Math.floor(window.innerWidth / 10))`
- Visibility API 暂停：`document.hidden` 时取消 AnimationFrame
- 使用 `will-change: transform` 提升合成层，避免主线程重绘

### 问题 2：Event Horizon 的滚动性能

50 天以上的"视界线"效果需要在每次滚动时计算视口内所有消息卡片到边缘的距离，应用动态模糊滤镜。直接遍历 DOM 读取 `getBoundingClientRect()` 会触发强制同步布局（Layout Thrashing）。

**优化策略**：
- 使用 `requestAnimationFrame` 防抖，限制为每帧最多执行一次
- 维护 `processingRef` 标志位防止重入
- 只对可见的消息卡片应用滤镜，而非全部

```typescript
const applyHorizon = useCallback(() => {
  if (processingRef.current) return; // 防重入
  processingRef.current = true;
  requestAnimationFrame(() => {
    // 批量读取 DOM → 批量写入样式
    const rects = bubbles.map(el => el.getBoundingClientRect());
    bubbles.forEach((el, i) => {
      const blurAmount = computeBlur(rects[i], viewport);
      if (blurAmount > 0.3) el.style.filter = `blur(${blurAmount}px)`;
    });
    processingRef.current = false;
  });
}, []);
```

### 问题 3：Gravity Lines 的 SVG 重计算

7 天以上需要在相邻不同发送者的消息卡片之间渲染 SVG `<line>` 连线。消息列表变化时（新消息到达、历史消息加载），需要重新计算所有锚点坐标。

**优化策略**：
- 仅当 `messageCount` 变化时触发重计算（React state 驱动）
- 使用 CSS `pointer-events: none` 让 SVG 层完全透明于交互
- 连线数量限制：最多渲染 20 对相邻不同发送者

## 架构决策：CSS-only vs Canvas vs WebGL？

| 效果 | 实现 | 理由 |
|------|------|------|
| 暗物质撤回 | CSS `backdrop-filter` | 纯视觉效果，GPU 合成，无 JS 开销 |
| 星尘粒子 | Canvas 2D | 40 个独立粒子，CSS animation 不够灵活 |
| 引力线 | SVG `<line>` | 需要根据 DOM 位置动态计算端点 |
| 潮汐锁定 | CSS `perspective` + `rotateY` | 纯 CSS 3D transform，浏览器原生优化 |
| 脉冲星光点 | CSS `animation` | 3 个固定光点纯 CSS 即可 |
| 微波背景辐射 | SVG `feTurbulence` | 程序化噪点，无需加载外部图片 |
| 红移/蓝移 | CSS `filter: sepia/hue-rotate` | 滚动事件驱动 CSS class 切换 |

关键原则：**能用 CSS 绝不用 JS，能用合成层绝不触发重排。**

## Docker 部署：如何让"宇宙"运行在容器里

Echo 的三容器架构（frontend/backend/db）在视觉系统上线后仍保持稳定：

```yaml
# docker-compose.prod.yml
frontend:
  build: ./frontend
  ports: ["8080:80"]
  # Nginx 静态服务，无状态，可水平扩展

backend:
  build: ./backend
  ports: ["9090:3001"]
  depends_on: { db: { condition: service_healthy } }
  # Express + Prisma，连续天数查询走 PostgreSQL

db:
  image: postgres:16-alpine
  ports: ["5433:5432"]
  # 存储用户、消息、Pair 连续天数原始数据
```

前端 Nginx 配置关键点：
```nginx
# 静态资源强缓存（带 hash 的 JS/CSS）
location /assets/ {
  expires 1y;
  add_header Cache-Control "public, immutable";
}

# HTML 不缓存（保证新版本即时生效）
location / {
  expires -1;
  add_header Cache-Control "no-cache";
}
```

## 总结

Echo 的"星轨共鸣"视觉系统是一次实验——把天体物理的时间尺度映射到社交关系上。从云原生的角度来看，它的核心经验是：

1. **状态隔离粒度 = 水平扩展粒度**：Pair 级别的隔离意味着任何请求都可以被路由到任何容器
2. **计算下沉到数据层**：连续天数的计算发生在数据库，而非应用内存
3. **CSS 是性能最好的渲染引擎**：能用 CSS 实现的视觉效果，GPU 会帮你处理一切
4. **DOM 操作要批处理**：read-write-read-write 模式是滚动性能的头号杀手
