# 从 0 到上线：Echo IM 的云端架构实践

## 项目背景

Echo IM 是一个面向移动端体验的即时通讯项目，核心功能包括好友聊天、群聊、语音消息、语音通话、朋友圈、背景自定义、表情包、会话搜索和关系型互动功能。这个项目不是单纯的前端 Demo，而是一个完整跑在云服务器上的 Web/App 混合应用。

作为云计算方向的学生，我最关注的不是“页面能不能写出来”，而是应用怎么部署、怎么通过 HTTPS 对外提供服务、怎么让 WebSocket 和实时通话在云环境里稳定工作。

## 整体架构

Echo 的生产环境采用一台 ECS 作为入口，容器化部署前端、后端和数据库：

- 前端：React + Vite，构建后由 Nginx 提供静态资源。
- 后端：Node.js + Express + Socket.IO，负责 REST API、实时消息和通话信令。
- 数据库：PostgreSQL，保存用户、好友、消息、朋友圈、群聊等数据。
- 文件存储：当前阶段使用服务器挂载目录保存上传文件，包括图片、语音、表情包、铃声。
- 访问入口：域名 `echo-im.cloud`，通过 HTTPS 暴露。

## 为什么要 HTTPS

聊天应用里很多能力都依赖安全上下文：

- 浏览器麦克风权限需要 HTTPS。
- PWA/App WebView 访问麦克风、通知等能力更依赖可信来源。
- WebRTC 通话在现代浏览器里基本要求 HTTPS。
- 用户登录 Token、图片和语音上传都不应该跑在明文 HTTP 上。

所以 Echo 的关键一步是把域名解析到 ECS，并通过 Nginx 做 HTTPS 入口，再反向代理到前端和后端容器。

## 云上部署链路

一次完整请求链路如下：

用户访问 `https://echo-im.cloud`，公网流量进入 ECS 的 Nginx。静态页面由前端容器返回，`/api` 请求转发到后端容器，`/socket.io` 连接转发到 Socket.IO 服务。后端再读写 PostgreSQL，并把上传文件写入持久化 volume。

这个结构的好处是边界清晰：

- 前端负责交互。
- 后端负责业务和权限。
- 数据库负责状态持久化。
- Nginx 负责入口、HTTPS、缓存和反向代理。
- Docker Compose 负责服务编排。

## 实时消息的云端问题

即时通讯最容易被忽略的是 WebSocket。在本地开发时，Socket.IO 连接通常很顺利；但上线后会遇到：

- HTTPS 页面不能连接 ws/http 混合内容。
- Nginx 需要正确支持 WebSocket Upgrade。
- 移动网络切换后连接可能断开，需要前端重连。
- 多端登录时需要处理在线状态和强制下线。

Echo 里通过 Socket.IO 维护在线状态、消息推送、输入中状态、语音通话信令。云端部署时，必须保证 `/socket.io/` 走同一个 HTTPS 域名，避免跨域和混合内容问题。

## 文件上传和云存储演进

当前版本文件上传保存在 ECS 的 Docker volume。这个方案适合早期项目：

- 成本低。
- 部署简单。
- 调试方便。

但它也有明显边界：

- 单机存储不适合水平扩容。
- 文件备份和迁移需要额外处理。
- 大量图片和语音会占用 ECS 磁盘。

后续更适合迁移到对象存储 OSS/COS/S3，把图片、语音、视频、表情包、铃声都放到云存储，并通过 CDN 加速访问。这样 ECS 只负责业务计算，静态资源交给云基础设施。

## 总结

Echo IM 的实践价值在于，它覆盖了云上应用常见链路：域名、HTTPS、反向代理、容器部署、数据库、文件上传、WebSocket、WebRTC。对于云计算学生来说，这比单纯写一个页面更接近真实项目，因为真正的难点往往出现在“上线之后”。
