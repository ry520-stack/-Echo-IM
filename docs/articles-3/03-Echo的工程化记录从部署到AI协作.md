# Echo 的工程化记录：从移动端白屏、Docker 部署到 AI 协作开发

Echo 做到后面，我发现真正花时间的地方不只是写功能，而是让项目在不同设备、不同浏览器、不同网络环境里都尽量稳定。

这篇记录一下 Echo 的工程化、部署、移动端兼容和 AI 协作开发。

项目地址：https://github.com/ry520-stack/-Echo-IM

## Docker 三容器部署

Echo 现在部署在自己的 ECS 上。

整体是三个容器：

- `echo-frontend-1`：Nginx 托管前端静态文件
- `echo-backend-1`：Express 后端服务
- `echo-db-1`：PostgreSQL 数据库

前端通过 Nginx 反向代理访问后端 API 和 Socket.IO。

这样做的好处是结构清楚：

- 前端只负责页面
- 后端只负责接口和实时通信
- 数据库独立保存数据
- Docker Compose 统一启动和更新

部署时也踩过不少坑，比如构建内存不足、环境变量缺失、上传目录 404、数据库 volume 不能误删等。

后来我给自己定了几条规则：

- 不删 `.env`
- 不删 uploads
- 不删数据库 volume
- 不执行 `docker system prune`
- 不执行 `docker volume prune`
- 不随便 `rm -rf`
- 先确认再清理

这些规则看起来很啰嗦，但对一个真实运行的项目很重要。

## 移动端白屏问题

最近修过一个问题：有些人打开网页是白屏，用 Via 浏览器才能打开。

这类问题通常不是一个原因，而是几类原因叠加：

- 手机自带浏览器 WebView 版本旧
- 前端构建产物用了旧浏览器不支持的语法
- 静态资源路径不兼容
- Service Worker 缓存了旧文件
- HTTP 环境下注册了不该注册的能力
- 加载失败时没有兜底提示，只剩白屏

最后我做了几件事：

- 加了 Vite legacy 构建，给旧浏览器提供兼容包
- 把 manifest、图标路径改成相对路径
- Service Worker 只在 HTTPS 下注册
- 启动阶段加兜底页面
- React 挂载失败时显示“重新加载/用系统浏览器打开”
- 通知权限只在安全上下文里请求

这样即使某些浏览器仍然不完全兼容，用户也不会只看到白屏。

## iPhone 和 HTTPS

苹果手机可以用 Safari 打开网页使用，但有些能力必须依赖 HTTPS。

例如：

- 摄像头
- 麦克风
- 通知
- PWA 能力
- Service Worker

如果只是用 IP + HTTP，基础页面可以打开，但很多系统能力会受限制。

所以后面域名备案和 HTTPS 配好之后，移动端体验会更完整。现在用 ECS IP 访问，适合测试基础功能；正式给别人使用，还是应该用域名和 HTTPS。

## 图片压缩和上传

移动端还有一个很常见的问题：图片太大。

手机原图动不动就是几 MB 到十几 MB，直接上传会导致：

- 上传慢
- 请求超时
- 手机浏览器卡顿
- 内存占用过高

Echo 前端用 `browser-image-compression` 做图片压缩，再上传到后端。这样可以减少移动端 OOM 和慢上传问题。

动态最多 18 张图，所以图片压缩非常重要。否则用户一次选很多图，浏览器很容易扛不住。

## 键盘弹出和页面高度

聊天软件在手机上还有一个很烦的问题：键盘弹出后，最新消息被挡住。

PC 上看不到这个问题，手机上非常明显。

Echo 里做了几类处理：

- 使用动态视口高度
- 键盘弹出后滚动到最新消息
- 监听 `visualViewport` 变化
- 输入栏固定在底部
- 给底部留安全距离

这类细节不是功能列表里最显眼的部分，但会直接影响用户觉得“能不能用”。

## 通知点击跳转

通知也踩过坑。

问题是：收到消息后弹出通知，点击通知页面刷新了，但没有进入对应聊天。

原因是路由 ID 不统一。有的地方用 Echo ID，有的地方用数据库 UUID，有的地方群聊又是 groupId。

后面我把通知点击逻辑改成携带明确 chatId，前端收到后跳到对应路由。并且在当前聊天页内，如果就是这个好友发来的消息，就不再弹顶部通知。

这类问题说明：通知不是单独功能，它和路由、会话、未读数、当前页面状态都有关。

## GitHub 清理和开源

开源前还做了一次清理。

不能把这些东西推上去：

- `.env`
- 数据库文件
- 证书
- `node_modules`
- `dist`
- uploads
- 本地调试目录

我最后是重新整理了 `.gitignore`，用干净目录重新初始化 Git 历史，再推到 GitHub。

这样仓库里只保留源码和文档，不带旧的敏感文件和构建产物。

## AI 协作开发的感受

Echo 这个项目很大一部分是和 AI 一起完成的。

用过的工具包括：

- Claude Code
- DeepSeek
- Mimo
- Codex

它们各有特点。

Claude Code 和 DeepSeek 适合快速写功能和改页面，Mimo 更像辅助理解，Codex 更适合长时间接管项目、读代码、改代码、部署和检查。

但无论哪个工具，都不是“你一句话，它自动做完所有事情”。

真正有效的方式是：

1. 把需求说清楚
2. 让 AI 先检查项目
3. 要求它给出修改方案
4. 让它实现
5. 自己测试
6. 把 bug 截图和现象反馈回去
7. 继续修

这个过程很像和一个工程师协作。

如果需求模糊，AI 会写出很多看似能运行但体验不对的东西。如果需求具体，AI 就能做得很稳定。

例如我会明确说：

- 不要删除 `.env`
- 不要删除数据库 volume
- 先列清单再清理
- 表情包默认 4 个，展开 12 个
- 动态图片最多 18 张
- 当前聊天页不要弹同一人的通知
- 删除会话刷新后不能回来
- 群聊左滑页叫群星域

这种要求越具体，结果越接近真实产品。

## 零基础也能做项目吗

我的感受是：现在 AI 工具确实让零基础或基础薄弱的人更容易从 0 到 1 做项目。

以前一个人想做全栈项目，需要同时学：

- 前端
- 后端
- 数据库
- 部署
- Linux
- Docker
- Nginx
- Git
- 报错排查

现在 AI 可以帮你把很多环节串起来。

但这不代表完全不用学。

你仍然需要知道自己想做什么，需要会判断结果对不对，需要能描述 bug，需要知道哪些东西不能乱删，需要不断测试。

AI 降低的是上手门槛，不是替代思考。

## 总结

Echo 让我最大的收获不是“做了一个聊天软件”，而是完整经历了一次从想法、开发、部署、测试、修 bug、开源、准备 App 打包的过程。

这个过程中我越来越感觉到：未来做项目，代码能力依然重要，但更重要的是把需求拆清楚，把问题描述清楚，把结果验证清楚。

AI 能帮你写很多代码，但项目能不能真正跑起来，还是取决于你能不能持续把它打磨下去。
