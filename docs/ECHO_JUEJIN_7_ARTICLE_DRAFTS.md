## 01. 我用 AI 从 0 搭了一个即时通讯项目：Echo 的第一版是怎么跑起来的

如果只是写一个聊天 Demo，其实很快。

一个输入框，一个消息列表，一个 WebSocket，十几分钟就能看起来像聊天软件。但真正开始做 Echo 之后，我发现“看起来能聊天”和“真的像一个产品”中间差了很多东西。

Echo 是我自己做的一个即时通讯项目。它最早的目标很简单：注册账号、添加好友、进入聊天、发送消息。后来慢慢扩展出了动态、星域权限、聊天背景、语音、图片、表情包、关系星域、5+ App 打包准备。

### 技术栈

前端我用了 React、Vite、TailwindCSS，主要考虑是开发快、组件拆分清晰、移动端样式调整比较直接。

后端是 Node.js + Express + Prisma，数据库用 PostgreSQL。实时通信使用 Socket.IO，部署在 ECS 上，通过 Docker Compose 管理前端、后端和数据库。

这个组合不算特别炫，但对一个个人项目来说足够务实：能快速迭代，能部署，出了问题也能查。

### 第一版先做闭环

我一开始没有急着做复杂 UI，而是先把最小闭环跑通：

- 用户注册和登录
- 好友搜索和添加
- 会话列表
- 私聊消息
- 在线状态
- 基础上传

这个阶段最重要的不是好看，而是确认项目结构能承受后面的功能增长。

如果一开始就堆动画、堆概念、堆页面，很容易做成一个漂亮但不可维护的壳。Echo 后面改了很多次，我越来越觉得：第一版的核心应该是“能稳定跑起来”。

### AI 参与了什么

这个项目里 AI 不是只帮我写一个组件，而是参与了很多轮：

- 生成初版页面
- 分析 Bug
- 修改后端接口
- 写部署命令
- 检查 ECS 和 Docker
- 生成文档
- 做代码审查

但 AI 不是自动驾驶。真正有用的是我把需求说清楚，比如“不能删数据库 volume”“不能删 .env”“删除会话刷新不能回来”“动态最多 18 张，一页 9 张左滑切页”。

需求越具体，AI 越像一个可靠工程师。需求越模糊，AI 越容易写出看似正确但体验不对的东西。

### 第一个教训

做聊天软件，第一天就要重视状态一致。

发送消息、接收消息、未读数、已读回执、撤回、删除、通知跳转，这些都不是孤立功能。它们会互相影响。

Echo 的第一版让我明白：即时通讯项目真正难的地方，不是把消息发出去，而是让每个端看到的状态都合理。

## 02. 聊天软件最难的不是发消息，而是状态一致

我修过一个很典型的问题：自己发一条消息，自己这边显示两条，但对方看起来正常。

这个问题乍一看很奇怪。后端是不是存了两条？Socket 是不是发了两次？数据库是不是重复写入？

最后发现，问题不在对方，而在发送方本地的状态合并。

### 乐观 UI 的代价

聊天软件通常会做乐观 UI：用户点发送后，前端先把消息插入列表，让体验显得很快。然后后端确认成功，再用真实消息替换临时消息。

这就带来一个问题：如果后端又把这条消息通过 socket 广播回发送方，前端就可能既保留临时消息，又追加真实消息。

结果就是：自己看到两条，对方只看到一条。

解决方式也很明确：发送方通过 ack 替换临时消息，后端不要再把同一条消息广播给发送者自己。

### 未读数为什么会残留

另一个问题是：明明进入了聊天，退出后会话列表仍然显示未读。

这类问题一般有两个原因：

第一，进入聊天时没有真正调用 mark read。

第二，mark read 被某个开关挡住了，比如“已读回执关闭”被误理解成“不清未读”。

我的处理方式是：清未读和是否通知对方已读要分开。用户进入会话，就应该清自己的未读；至于要不要告诉对方“我已读”，再受已读回执设置控制。

### 删除会话刷新又回来

会话列表长按删除后消失，刷新又回来，这也是状态一致问题。

如果只是前端把会话从数组里过滤掉，那刷新后后端还是会根据历史消息把会话返回。

所以需要后端记录一个 clear time。获取会话列表时，只看 clear time 之后的新消息。如果清空之后没有新消息，就不要再返回这个会话。

### 通知点击进错页面

还有一个问题：收到消息通知，点击后页面刷新了，但没有进入对应聊天。

根因是路由参数不统一。有的地方用数字 Echo ID，有的地方用数据库 UUID。聊天页搜索只支持一种时，通知跳转就可能找不到人。

最后我让通知优先使用 digitalId，同时后端搜索兼容 UUID 兜底。

### 结论

聊天软件不是“发消息”这么简单。

每个功能都在维护一件事：不同页面、不同设备、不同状态下，用户看到的信息不能互相打架。

这也是我做 Echo 后最大的感受：IM 的复杂度藏在边界里。

## 03. Echo 的“星域”：我不想只做朋友圈权限

很多社交软件都有好友分组。

但我一直觉得“分组”这个词太工具化了。它只是把人放进不同文件夹，没有表达关系，也没有表达权限。

所以在 Echo 里，我把好友分组叫做“星域”。

### 星域不是装饰

Echo 的星域不是为了听起来高级，而是直接对应权限。

比如：

- 密友星域：可以看更私密的动态
- 同学星域：可以看普通动态
- 同事星域：只能看公开动态
- 陌生星域：默认不可见

这样“你把谁放在哪个星域里”，就不只是整理通讯录，而是在定义关系边界。

### 权限模型

我最后比较喜欢的模型是：

最终权限 = 星域允许 - 单独排除

也就是说，我可以发一条动态，只给“密友”和“同学”看，但单独排除某个人。

这比单纯的“谁可见 / 谁不可见”更适合长期维护。因为人多以后，一个个选人会非常痛苦，而按星域组织会更自然。

### 关系星域

Echo 还有一个“关系星域”页面。

它不是好友资料页，也不是朋友圈页，而是你和某个人之间的关系页。

里面可以显示：

- 对方头像、昵称、在线状态
- 所属星域
- 认识时间
- 回声值
- 最近连接
- 对方最近动态

聊天页左滑进入关系星域，这个设计是为了避免把所有功能都塞进聊天页。

聊天页负责即时沟通，关系星域负责关系理解，资料页负责设置。

### 为什么这很重要

如果一个社交产品只有聊天，它很容易变成工具。

我希望 Echo 不只是能发消息，而是能表达关系层次。谁和你更近，谁能看到更多，谁只是普通联系人，这些都应该被产品结构承载。

这就是星域的意义。

## 04. 动态图片为什么要做两种展示：悬浮折叠和宫格分页

动态图片展示看起来是个小功能，但移动端体验差异很大。

一开始图片横向展示，看起来很直观，但实际用起来很慢。尤其图片多的时候，需要一直左右找，效率不高。

后来我给 Echo 做了两种模式：悬浮折叠和宫格展示。

### 悬浮折叠

悬浮折叠适合少量图片。

它像一叠卡片，当前图片在最上层，后面的图片露出一点层次。用户左右滑可以切换。

这种模式更有情绪感，也更适合聊天和动态里的“随手分享”。

### 宫格展示

宫格展示适合多图。

Echo 现在动态最多支持 18 张图。宫格模式每页 9 张，3x3 展示。如果超过 9 张，第 10 张会进入第二页，用户左滑切换。

这个设计有几个好处：

- 一眼看到更多图片
- 不需要横向慢慢找
- 页面高度可控
- 符合用户对相册/朋友圈的直觉

### 发动态预览也要宫格

之前发动态时，上传的图片也是横向滚动。这个体验很差，因为用户发 10 多张图时，很难知道自己选了哪些。

所以我把发动态预览也改成 3 列宫格，最多 18 张。

发布时还会保存展示模式和封面序号。这样别人看到的动态，不会因为本机 localStorage 不同而显示不一致。

### 一个小结论

图片体验不是图片越大越好，也不是动画越多越好。

少图要有氛围，多图要有效率。

Echo 这次动态图片改造，本质就是在“好看”和“好用”之间找平衡。

## 05. 从网页到 5+ App：拍照、录音、通话为什么不能只靠 Web

做 Echo 的时候，我踩到一个很现实的问题：网页和 App 的能力边界不一样。

在电脑浏览器里看起来没问题的功能，到了手机网页上就可能变得很奇怪。

### 拍照和相册

网页里可以用 `<input type="file" accept="image/*" capture>` 尝试唤起相机。

但不同手机、不同浏览器行为不一致。有的会打开相册，有的会让你选择项目，有的会受 HTTPS 限制。

如果想更像微信那样直接拍照、拍视频，5+ App 里可以用 `plus.camera` 调原生相机。

### 录音

网页录音依赖 `MediaRecorder` 和麦克风权限。

在 http、非安全上下文、部分手机浏览器里，录音可能直接不可用。

5+ App 里可以用 `plus.audio.getRecorder()`，这更接近原生 App 的能力。

### 通话

实时语音通话更复杂。

WebRTC 需要安全上下文，也就是 HTTPS 或 localhost。公网稳定通话还需要 STUN/TURN。只有 STUN 的情况下，某些网络能通，某些网络就不行。

所以“能打电话”和“稳定像微信一样打电话”不是一个级别。

### 通知

网页通知也有边界。

页面开着时，可以用 socket + 页面内通知。页面隐藏时，可以尝试浏览器 Notification。但如果浏览器被杀后台，就不稳定。

真正的网页后台通知需要 Web Push、Service Worker、VAPID。5+ App 则应该接 UniPush/个推或厂商推送。

### 我的结论

HBuilderX 5+ App 不是简单“网页套壳”四个字能概括。

它确实加载网页，但也能通过 plus API 调用原生能力。真正要做好，就要把 Web 能力和原生能力分层设计。

## 06. 个人项目上线后，真正折磨我的是 Docker 和磁盘

很多人做个人项目，只关注代码能不能跑。

但项目一旦上线，真正麻烦的东西就来了：服务器、Docker、数据库、上传文件、磁盘空间、备份和清理。

Echo 部署在 ECS 上，前端、后端、PostgreSQL 都用 Docker Compose 管理。

### 什么不能乱删

聊天项目最怕误删数据。

我给自己定了几条规则：

- 不删 `.env`
- 不删 uploads
- 不删数据库 volume
- 不删正在运行的容器
- 不执行 `docker system prune`
- 不执行 `docker volume prune`
- 不执行数据库重置

这些规则看起来麻烦，但能救命。

### Docker 镜像会越来越多

每次重新构建后端，都会留下旧镜像。Echo 的后端镜像大约 1.56GB，一个两个没感觉，多了以后磁盘很快就上去。

我检查服务器时发现根分区已经 88%。

这时候卡顿不一定是代码问题，磁盘压力也会影响 Docker、数据库、上传和构建速度。

### 清理要精确

我不建议个人项目上来就 `docker system prune`。

更稳的做法是：

- 先列出镜像、容器、volume
- 确认哪些是 dangling image
- 确认不是当前运行镜像
- 按镜像 ID 精确删除

这比一条 prune 慢，但安全很多。

### 运维也是产品的一部分

一个聊天 App 如果上传图片丢了，数据库没了，用户账号没了，那前端做得再漂亮也没有意义。

所以我现在越来越觉得：个人项目上线后，运维纪律和代码质量一样重要。

## 07. 从 Claude Code + DeepSeek/Mimo 到 Codex：我的 AI 协作开发心得

Echo 这个项目里，我用了不止一种 AI 开发方式。

前期我尝试过 Claude Code 接 DeepSeek 和 Mimo，让它们按文档做功能。后来我更多使用 Codex 来做代码检查、修 Bug、部署和整理项目。

这段经历让我对 AI 写代码的看法变得更现实。

### AI 很快，但不等于一定对

AI 写代码速度很快，尤其是搭页面、补接口、生成 CRUD。

但它容易出现两个问题：

第一，看起来做完了，实际上只做了表面。

第二，改一个地方时没有理解整个项目的状态流。

比如聊天项目里，发消息、会话列表、通知、未读、已读、删除会话是连在一起的。如果只修一个点，很可能另一个点又坏。

### 文档非常重要

我后来发现，和 AI 协作最重要的不是提示词多华丽，而是文档和边界清楚。

例如我会明确写：

- 不准删 `.env`
- 不准删 uploads
- 不准动 Docker volume
- 不准直接 prune
- 每次改完要构建
- 部署后要检查容器
- 功能要按用户体验验收

这些约束能显著降低 AI 乱改的风险。

### DeepSeek/Mimo 和 Codex 的分工

我的感受是：

DeepSeek/Mimo 更适合快速生成方案、做初版页面、铺功能。

Codex 更适合在已有代码里做工程化修复：查文件、看 diff、改局部、跑构建、连服务器、部署、复盘。

当然这不是绝对的，但对我这个项目来说，这种分工比较顺。

### 人要负责产品判断

AI 可以告诉我“这个功能能这样做”，但它不能替我决定 Echo 应该是什么。

比如“星域”这个概念，不是 AI 自动想出来就完事。它要和好友分组、动态权限、关系页面、资料设置统一起来。

人必须负责产品方向、体验判断和最终验收。

### 最后

AI 开发不是“我不用写代码了”。

更准确地说，它让一个人可以承担以前需要多人协作的工作：产品、前端、后端、运维、文档、测试。

但前提是你要清楚自己要什么，也要敢于检查它做得对不对。

Echo 还没有完成，但这个项目已经让我确认了一件事：未来个人开发者的能力边界会被 AI 拉大很多。
