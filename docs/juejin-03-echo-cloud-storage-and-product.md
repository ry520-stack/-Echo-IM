# Echo IM 的云端产品化：上传、搜索、个性化与下一步演进

## 从聊天工具到产品体验

Echo IM 不只是“能发消息”。为了让它更像一个完整产品，项目逐步加入了：

- 会话列表搜索。
- 好友 ID、备注、原网名搜索。
- 聊天记录搜索。
- 图片、语音、表情包上传。
- 朋友圈动态。
- 自定义聊天背景。
- 自定义来电铃声。
- 群聊和拉群。
- 双人共同宠物。

这些功能背后都涉及云端数据建模和资源管理。

## 上传能力的设计

Echo 目前支持多类上传：

- 头像。
- 聊天图片。
- 语音消息。
- 视频。
- 表情包。
- 朋友圈图片。
- 自定义铃声。
- 背景图。

上传功能看起来简单，但云上要考虑很多细节：

- 文件大小限制。
- MIME 类型校验。
- Nginx 请求体大小。
- 后端上传目录持久化。
- App WebView 下本地文件路径上传。
- 图片压缩质量。
- GIF 不应被错误压缩。
- 上传失败时返回 JSON，而不是 HTML 错误页。

其中“Unexpected token '<'”就是典型问题：前端以为后端返回 JSON，实际收到的是 Nginx 或错误页面返回的 HTML。解决方式是前端检查 `content-type`，后端提供明确上传接口，Nginx 保证请求能正确代理。

## 搜索能力的产品价值

IM 产品里搜索非常重要。Echo 的搜索包括两层：

- 搜索好友：Echo ID、备注、昵称、原网名。
- 搜索聊天记录：关键词、指定聊天、日期。

从架构上看，搜索不应该只在前端本地过滤，因为用户历史消息可能很多，也可能分页加载。更稳的做法是后端提供搜索接口，按用户权限过滤，再返回匹配结果。后续如果消息量增大，可以接入 Elasticsearch 或 PostgreSQL 全文索引。

## 个性化设置如何云端化

Echo 里有很多个性化设置：

- 聊天背景。
- 会话列表背景。
- 朋友圈背景。
- 来电铃声。
- 高亮颜色。
- 已读回执。

有些设置适合本地保存，比如界面高亮颜色；有些设置必须账号级保存，比如来电铃声。因为别人给我打电话时，播放的是我的铃声，这个信息需要对方发起通话时从服务器获取。

这就是产品逻辑决定数据归属：只影响自己当前设备的可以本地化；会影响别人或跨设备一致性的必须云端化。

## 共同宠物的云端模型

Echo 的共同宠物不是每个人一只，而是一段关系一只。第一版先做成聊天里的“第三个成员”：它偶尔发短句，像一个轻量陪伴者。

后续可以扩展：

- `PetBond` 表记录两人关系里的宠物。
- 记录等级、亲密度、连续互动天数。
- 结合最近少量上下文生成台词。
- 接入大模型，但不上传完整聊天记录。
- 用规则系统限制输出长度、语气和安全边界。

这个功能很适合体现“云 + AI + 社交产品”的结合：云端保存关系状态，AI 负责生成轻量表达，前端负责自然呈现。

## 下一步云架构升级

如果 Echo 继续产品化，可以按下面方向演进：

- 文件迁移到对象存储。
- 图片和视频接 CDN。
- TURN 服务独立部署。
- Redis 保存在线状态和 Socket.IO 横向扩展。
- 消息表分区或归档。
- 日志接入云监控。
- 自动化 CI/CD 发布。
- 数据库定期备份。

## 总结

Echo IM 是一个很适合云计算学生展示能力的项目。它既有前端交互，也有后端业务；既有数据库，也有文件上传；既有 HTTPS 和 WebSocket，也有 WebRTC 和移动端适配。真正的价值不只是“做了聊天”，而是把一个聊天产品完整跑在云上。
