# Echo 抖音脚本包

目标：用短视频连续介绍 Echo 项目、开发过程、AI 协作、踩坑心得。

项目地址：https://github.com/ry520-stack/Echo-IM

账号主页建议：简介里可以写“一个人做 Echo IM，记录从网页到 App 的全过程”，GitHub 地址放在主页可见位置或评论区置顶。

## 账号内容定位

人设：一个真实做项目的个人开发者。

风格：
- 不装专家。
- 多用手机录屏。
- 讲真实问题，不讲空话。
- 每条视频只讲一个点。
- 让观众感觉“这个项目确实在做，不是 PPT”。

## 视频 1：我一个人做了个聊天 App，叫 Echo

时长：45-60 秒

开头：
“这是我自己做的聊天 App，名字叫 Echo。它现在已经能注册、加好友、聊天、发动态了。”

画面：
- 手机打开 Echo。
- 展示会话列表。
- 进入聊天。
- 发一条消息。
- 左滑进入关系星域。

口播：
“一开始我只是想做一个简单聊天工具，但做着做着发现，聊天软件最难的不是发消息，而是状态。比如未读、已读、撤回、删除会话、通知跳转、在线状态，每一个都可能出 Bug。”

中段：
“Echo 里我还做了一个叫星域的设计，它不是普通好友分组，而是和动态权限绑定。你可以理解成不同关系圈层。”

结尾：
“后面我会继续记录这个项目怎么从网页一步步变成 5+ App。”

字幕关键词：
个人开发 / 即时通讯 / Echo / 星域 / AI 协作

## 视频 2：聊天软件最烦的 Bug：自己发一条，自己显示两条

时长：30-45 秒

开头：
“你以为聊天软件只要能发出去就行？我遇到过一个 Bug：我发一条，自己这边显示两条，但对方正常。”

画面：
- 聊天页发消息。
- 屏幕上标注“临时消息”和“真实消息”。

口播：
“原因是前端做了乐观 UI，点发送后先把临时消息插到列表里。后端确认后又通过 socket 把真实消息发回来，前端没有替换，而是追加，所以自己看到两条。”

解决：
“最后我让发送方只通过 ack 替换临时消息，后端不再把同一条消息广播给发送者自己。”

结尾：
“IM 的坑很多都不是功能不会做，而是状态没对齐。”

## 视频 3：我为什么把好友分组叫“星域”

时长：45 秒

开头：
“Echo 里我没有把好友分组叫分组，而是叫星域。”

画面：
- 星域管理页。
- 好友设置页里的星域分组。
- 发动态时选择可见星域。

口播：
“因为我不想它只是一个通讯录文件夹。星域在 Echo 里对应权限，比如密友星域能看私密动态，同事星域只能看公开动态。”

重点：
“最终权限可以理解成：星域允许，再单独排除某些人。”

结尾：
“我希望 Echo 的关系系统不是冰冷的分组，而是真的能表达关系边界。”

## 视频 4：动态图片为什么最多 18 张

时长：45 秒

开头：
“我把 Echo 动态图片上限改成了 18 张，不是随便改的。”

画面：
- 发动态上传 18 张图片。
- 宫格 3x3。
- 左滑到第二页。
- 悬浮折叠模式。

口播：
“少图的时候，我用悬浮折叠，看起来像一叠卡片。多图的时候，用宫格展示，一页 9 张，最多两页。”

重点：
“第 10 张不会挤在横向滚动里，而是进入第二页，左滑切换。”

结尾：
“移动端图片体验，关键不是炫，而是让用户找图不累。”

## 视频 5：网页为什么不能像 App 一样稳定拍照、录音、打电话

时长：60 秒

开头：
“我做 Echo 时发现，网页和 App 的能力边界真的不一样。”

画面：
- 网页点拍照。
- 录音按钮。
- 通话按钮。
- HBuilderX/5+ App 概念图或代码截图。

口播：
“网页可以用 input 调相册或相机，但不同手机浏览器表现不一样。录音和通话还要求 HTTPS。WebRTC 要稳定公网通话，还需要 TURN。”

5+ App：
“如果用 HBuilderX 打包成 5+ App，就可以通过 plus.camera、plus.audio 调原生拍照和录音。”

结尾：
“所以不是网页套壳没用，而是要把 Web 能力和原生能力分清楚。”

## 视频 6：AI 写代码靠谱吗？我让 DeepSeek、Mimo、Codex 都参与了

时长：60 秒

开头：
“Echo 这个项目不是我一个字一个字手写完的，我用了 AI 协作。”

画面：
- 文档截图。
- 代码 diff。
- 终端构建。
- ECS 部署。

口播：
“前期我试过 Claude Code 接 DeepSeek 和 Mimo，让它们按文档做功能。后来我更多用 Codex 查代码、修 Bug、部署和整理项目。”

重点：
“我的感受是，AI 很快，但不能放任。你必须告诉它不能删什么、改完怎么验收、部署要检查什么。”

结尾：
“AI 不是替你做产品决定的人，它更像一个执行力很强的工程师。”

## 视频 7：个人项目上线后，Docker 镜像差点把服务器吃满

时长：45 秒

开头：
“做个人项目上线后，我发现最烦的不一定是代码，而是服务器磁盘。”

画面：
- ECS 磁盘 88% 截图。
- docker images 里很多 `<none>`。
- 后端镜像 1.56GB。

口播：
“Echo 每次构建后端，都会留下旧镜像。一个后端镜像 1.56GB，多构建几次，磁盘很快就上去了。”

重点：
“但是聊天项目不能乱 prune，不能删 volume，不能碰 uploads，不然数据可能没了。”

结尾：
“个人项目上线后，运维纪律真的很重要。”

## 视频 8：Echo 下一步我要做什么

时长：45 秒

开头：
“Echo 现在还不是最终版，我下一步准备做这几件事。”

画面：
- 项目功能页快速切换。
- 通知、通话、星域、动态画面。

口播：
“第一，等域名下来后做 HTTPS。第二，把 Web Push 和 5+ App 原生通知接起来。第三，补 TURN，让语音通话更稳定。第四，继续优化移动端性能。”

结尾：
“我会继续把这个项目真实开发过程发出来，包括成功的，也包括踩坑的。”

## 拍摄建议

- 每条视频开头 3 秒必须给结果，不要先讲背景。
- 屏幕录制要放大关键区域，比如按钮、聊天气泡、星域卡片。
- 多用字幕标注：Bug 原因、解决方案、踩坑点。
- 不要一条视频讲完整项目，拆开讲。
- 不要说“吊打微信”这种容易招黑的话，定位成个人项目探索。

## 可反复使用的结尾话术

- “这个项目我会继续迭代，下一条讲具体怎么修。”
- “如果你也在做个人项目，这个坑可以提前避开。”
- “Echo 还没完成，但我会把它从网页一步步做到 App。”
- “AI 能帮你写代码，但产品判断还是得自己做。”
