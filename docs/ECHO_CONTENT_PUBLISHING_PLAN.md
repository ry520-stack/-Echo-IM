# Echo 内容发布策划

更新时间：2026-05-24

项目地址：https://github.com/ry520-stack/Echo-IM

发布建议：掘金文章末尾可以放“项目持续迭代中，仓库地址：...”这一句；抖音评论区或主页简介可以放 GitHub 地址，方便别人看到真实项目。

## 一句话定位

Echo 是一个从网页即时通讯开始，逐步走向 5+ App 的个人社交产品实验。它不是只做“能发消息”，而是把聊天、动态、关系权限、好友分组、背景个性化、语音通话和 AI 协作开发都揉进一个真实项目里。

## 掘金 7 篇系列文章

### 1. 我用 AI 从 0 搭了一个即时通讯项目：Echo 的第一版是怎么跑起来的

角度：项目起源、技术栈、最小闭环。

要讲：
- 为什么想做 Echo，而不是再写一个普通 Chat Demo。
- 前端 React + Vite + Tailwind，后端 Node/Express + Prisma + PostgreSQL。
- Socket.IO 如何支撑实时消息、在线状态、输入中、已读回执。
- 第一个能跑版本的关键：先保证注册、登录、好友、聊天闭环。

适合标题：
- 《我用 AI 从 0 搭了一个即时通讯项目，第一步不是写聊天框》
- 《一个人做社交 App：Echo 的第一个可用版本》

### 2. 聊天软件最难的不是发消息，而是“状态一致”

角度：消息系统稳定性。

要讲：
- 自己发消息出现两条的原因：乐观 UI + socket 回包重复。
- 对方正常、自己异常，说明问题在本地消息合并。
- 未读数为什么会残留：进入会话后是否真正 mark read。
- 长按删除会话刷新又回来：只删前端状态不够，后端会话列表也要尊重 clear time。
- 通知点击无法进入聊天：路由参数 digitalId / UUID 不一致。

适合标题：
- 《做聊天软件后我才知道：消息发出去只是开始》
- 《我修了 10 个聊天 Bug，才明白 IM 的状态一致有多难》

### 3. Echo 的“星域”：我不想只做朋友圈权限

角度：产品设计和权限模型。

要讲：
- 传统好友分组太工具化，Echo 把分组叫“星域”。
- 星域不是装饰，它对应动态可见权限。
- 最终权限 = 星域允许 - 单独排除。
- 好友资料页、关系星域页、星域管理页的职责划分。
- 为什么左滑聊天进入“与某人的星域”，而不是塞一堆设置在聊天页。

适合标题：
- 《我给聊天 App 做了一个“星域”系统：分组不是分组，是权限》
- 《不想再做普通朋友圈权限，于是我设计了 Echo 星域》

### 4. 动态图片为什么要做两种展示：悬浮折叠和宫格分页

角度：移动端图片交互。

要讲：
- 动态图片不是简单 `<img>` 列表。
- 悬浮折叠：适合少图、情绪化、像卡片一样浏览。
- 宫格展示：适合多图，一页 9 张，最多 18 张，左滑切页。
- 发动态上传预览为什么改成 3 列宫格。
- 图片上传、压缩、懒加载、预览大图、封面选择的权衡。

适合标题：
- 《动态发 18 张图怎么不卡？我给 Echo 做了两套图片展示》
- 《从横向滚动到 3x3 宫格：移动端图片体验的一次重构》

### 5. 从网页到 5+ App：拍照、录音、通话为什么不能只靠 Web

角度：Web 和 App 能力边界。

要讲：
- http 下网页不能稳定录音、拍照、通话，HTTPS 是基础。
- 浏览器 `input capture` 和真正原生相机不是一回事。
- 5+ App 虽然像网页套壳，但可以通过 `plus.camera`、`plus.audio` 调原生能力。
- WebRTC 通话需要 HTTPS、STUN/TURN，公网稳定通话不能只靠本地测试。
- 后台通知更不能靠网页弹窗，Web 要 Web Push，App 要 UniPush/厂商推送。

适合标题：
- 《网页套壳就能做 App 吗？我做 Echo 后踩到的移动端能力边界》
- 《拍照、录音、通话、通知：Web 和 5+ App 的真实差距》

### 6. 一个人开发一个社交 App，运维和部署比想象中重要

角度：ECS、Docker、数据库、上传目录。

要讲：
- 本地 Windows 项目 + ECS Linux 部署的协作方式。
- Docker Compose 管前端、后端、PostgreSQL。
- 为什么不能随便 `docker system prune`、不能删 volume、不能碰 uploads。
- 清理旧镜像、旧压缩包、dist、node_modules 的原则。
- 线上卡顿不一定是代码，磁盘 88% 也会拖慢构建和服务。

适合标题：
- 《个人项目上线后，真正折磨我的是 Docker 和磁盘》
- 《别乱 prune：一个聊天项目的 ECS 部署和清理原则》

### 7. 从 Claude Code + DeepSeek/Mimo 到 Codex：我的 AI 协作开发心得

角度：AI 协作开发方法论。

要讲：
- 一开始用 Claude Code 接 DeepSeek/Mimo，让 AI 按文档做功能。
- AI 能写代码，但经常需要人定义产品边界和验收标准。
- 后来用 Codex 做代码审查、修 Bug、部署、整理文档。
- 最有效的模式：人负责方向和体验，AI 负责实现、排查、文档化。
- 不要把 AI 当神，也不要只让 AI “随便改”。要给禁止项、验收项、部署边界。

适合标题：
- 《我用 Claude Code、DeepSeek、Mimo、Codex 合作开发了一个聊天 App》
- 《AI 写代码很快，但真正救项目的是验收标准》

## 抖音内容主线

### 人设

一个真实做项目的人，不装大厂专家，不讲空泛概念。重点是“我真的把它做出来了，也真的踩坑了”。

表达风格：
- 口语化。
- 多展示手机录屏。
- 少说框架名，多说问题和结果。
- 每条视频只讲一个点。

### 10 条短视频选题

1. 我一个人做了个聊天 App，名字叫 Echo。
2. 这个聊天框不是 Demo，真的能注册、加好友、发消息。
3. 为什么我把好友分组叫“星域”？
4. 聊天软件最烦的 Bug：自己发一条，自己显示两条。
5. 我做动态图片展示，最后选择了 18 张上限。
6. 网页为什么不能像 App 一样稳定拍照、录音、打电话？
7. 我准备用 HBuilderX 把 Echo 打包成 5+ App。
8. AI 写代码到底靠不靠谱？我让 DeepSeek/Mimo/Codex 都参与了。
9. 个人项目上线后，Docker 镜像把服务器磁盘快吃满了。
10. Echo 下一步：通知、通话、App 打包、星域权限继续完善。

## 个人心得主线

可以反复讲这几个观点：

- AI 不是替你做产品决定的人，它更像一个执行力很强的工程师。
- 需求越清楚，AI 写出来越接近你想要的东西。
- 不要一次让 AI 改太多，改完必须验收。
- 真正难的是连续迭代，不是第一版能跑。
- 做聊天 App 会逼你理解状态一致、权限、媒体能力、部署和移动端限制。
- 一个人做项目最重要的是保留文档，否则过几天自己也会忘。

## 发布顺序建议

1. 先发抖音第 1 条，建立项目存在感。
2. 掘金发第 1 篇，讲完整项目背景。
3. 抖音发聊天 Bug、星域、动态图片这几个短点。
4. 掘金第 2、3、4 篇跟上，形成技术深度。
5. 域名和 5+ App 有进展后，再发第 5 篇。
6. 部署清理和 AI 协作放后面，更适合有项目上下文后发表。
