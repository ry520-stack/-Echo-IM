# 我用 AI 从 0 到 1 做了一个即时通讯项目：Echo

最近我在学习做项目，用 Claude Code、DeepSeek、Mimo 和 Codex 一起折腾了一个即时通讯项目，名字叫 Echo。

一开始它只是一个很简单的想法：能不能自己做一个可以登录、加好友、聊天的小应用。后来慢慢加上了动态、星域分组、关系页、表情包、背景设置、群聊和 5+App 打包适配，项目也从一个聊天 Demo 变成了一个更完整的移动端 IM 产品雏形。

项目地址：https://github.com/ry520-stack/-Echo-IM

## 为什么做 Echo

我最开始做项目，是想让简历上有一个真正能跑起来的东西。

但做着做着会发现，如果只是写一个“看起来像聊天软件”的页面，其实并不难。一个输入框、一个消息列表、一个发送按钮，再接一个 WebSocket，就能做出最基础的聊天效果。

真正难的是：它能不能像一个产品一样使用。

比如：

- 登录状态要不要保持？
- 发出去的消息会不会重复？
- 对方在线状态准不准？
- 刷新后未读数会不会还在？
- 删除会话后会不会刷新又回来？
- 手机键盘弹出来后，最新消息会不会被挡住？
- 图片太大会不会把手机浏览器卡死？
- 手机自带浏览器、Safari、微信内置浏览器能不能打开？

这些问题都不是“写一个聊天框”能解决的。

所以 Echo 后来真正做的事情，是把一个个边界场景慢慢补齐。

## 当前 Echo 有什么

目前 Echo 已经实现了这些基础能力：

- 登录、注册、找回密码
- Echo ID 搜索用户
- 好友申请、同意、拒绝、删除、拉黑
- 私聊消息实时收发
- 图片、语音、视频上传接口
- 消息撤回、删除、清空会话
- 已读回执、未读数、在线状态
- 会话列表搜索好友和聊天记录
- 动态发布、点赞、评论、删除
- 动态图片悬浮折叠和宫格展示
- 表情包上传、管理、展开分页
- 星域分组和动态可见权限
- 好友关系星域页
- 群聊、群成员、群头像、群公告、群昵称
- 会话列表背景、聊天背景、引力圈背景
- 5+App / HBX 打包适配准备

它还不是一个完全成熟的 App，但已经不只是“能发消息”的 Demo 了。

## 技术栈

前端主要是：

- React
- TypeScript
- Vite
- Tailwind CSS
- Framer Motion
- React Router
- Socket.IO Client

后端主要是：

- Node.js
- Express
- TypeScript
- Prisma
- PostgreSQL
- Socket.IO
- Multer
- JWT

部署上用了：

- Docker Compose
- Nginx
- PostgreSQL
- 阿里云 ECS

现在前端、后端和数据库都跑在自己的 ECS 上。前端由 Nginx 托管，后端提供 API 和 Socket.IO，数据库用 PostgreSQL。

## 第一个阶段：先跑通闭环

做 Echo 的第一步不是把界面做得多漂亮，而是先跑通最小闭环：

- 用户能注册登录
- 能搜索到好友
- 能发送好友申请
- 能进入会话
- 能发送和接收消息
- 能刷新后继续使用

这个阶段最重要的是“项目结构能不能撑住后续迭代”。

如果一开始就疯狂堆 UI、动画、概念，很容易做出一个看起来不错但无法维护的壳。Echo 后面改过很多轮，我越来越觉得，第一版最重要的是稳定跑起来。

## IM 项目真正麻烦的地方

做聊天软件之后，我最大的感受是：IM 的复杂度藏在状态里。

发消息只是表面。

真正麻烦的是：

- 发送方本地先显示消息，后端确认后怎么替换？
- Socket 推送回来时，会不会把自己的消息再加一遍？
- 已读回执和未读数是不是两个不同概念？
- 删除会话是前端过滤，还是后端记录清空时间？
- 通知点击后应该跳到 Echo ID 还是数据库 ID？
- 当前就在聊天页时，还要不要弹通知？

这些问题任何一个没处理好，用户都会觉得“这个软件怪怪的”。

例如我修过一个问题：自己发一条消息，自己这边显示两条，但对方显示正常。

原因不是数据库存了两条，而是前端做了乐观更新，同时又接收到了后端广播回来的同一条消息。解决方式是：发送方用 ack 替换临时消息，不再重复追加。

这类问题很小，但非常影响体验。

## AI 在项目里做了什么

这个项目不是我一个人纯手写出来的。Claude Code、DeepSeek、Mimo 和 Codex 都参与过不同阶段。

AI 帮我做过这些事情：

- 生成初版页面
- 分析 bug 原因
- 修改前端交互
- 补后端接口
- 检查 Docker 和 ECS
- 写部署文档
- 做代码审查
- 整理发布文章

但我也明显感觉到，AI 不是自动驾驶。

你给一个模糊需求，它可能写出一个看似正确但体验不对的功能。你把需求说清楚，比如“不能删 .env”“不能删数据库 volume”“删除会话刷新后不能回来”“表情包默认 4 个，展开 12 个，支持分页”，它才更像一个可靠的工程助手。

所以现在做项目，重要的不只是会不会写代码，而是能不能把需求拆清楚、说清楚、验证清楚。

## 总结

Echo 让我最大的感受是：现在的 AI 编程工具确实把开发门槛降低了很多。

以前从 0 到 1 做一个全栈项目，需要自己懂前端、后端、数据库、部署、Docker、服务器和各种报错。对基础薄弱的人来说，光环境配置就可能卡很久。

但现在，只要愿意不断描述需求、测试结果、反馈 bug，再让 AI 一轮轮修改，确实可以把一个项目搭起来。

Echo 还会继续更新。后面我会继续记录它从 Web 版到 5+App、从功能可用到体验稳定的过程。
