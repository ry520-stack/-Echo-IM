# Echo IM 实时通信实践：Socket.IO、WebRTC 与云网络

## 为什么 IM 项目适合练云计算

即时通讯项目天然依赖网络能力。普通 CRUD 项目更多是请求响应，而 IM 系统需要长期连接、实时推送、音频采集、通话信令和弱网恢复。这些能力上线到云服务器后，会直接暴露网络、协议、安全和部署问题。

Echo IM 的实时能力主要分为三层：

- 消息实时同步：Socket.IO。
- 语音消息：录音、上传、播放。
- 语音通话：WebRTC + Socket.IO 信令。

## Socket.IO 在项目里的职责

Echo 后端用 Socket.IO 承担这些任务：

- 用户上线/离线状态广播。
- 私聊消息实时推送。
- 群聊消息实时推送。
- 输入中状态。
- 语音通话邀请、接受、拒绝、挂断。
- WebRTC offer、answer、candidate 信令转发。

Socket.IO 不保存长期数据，真正的消息记录会落库到 PostgreSQL。这样可以保证用户刷新页面、换设备登录后仍然能拉取历史记录。

## WebRTC 通话为什么需要信令服务器

WebRTC 负责浏览器之间的音频传输，但它本身不负责“找人”和“协商”。所以 Echo 使用 Socket.IO 做信令通道：

1. A 点击语音通话。
2. 后端检查 A 和 B 是否为好友。
3. 后端向 B 推送 `call:invite`。
4. B 接听后，A 创建 offer。
5. B 返回 answer。
6. 双方交换 ICE candidate。
7. 音频流建立点对点连接。

这个过程中，Socket.IO 只是“牵线”，真正的音频数据走 WebRTC。

## HTTPS 对通话的影响

移动端和浏览器对麦克风权限非常严格。Echo 早期遇到过“当前环境不支持麦克风通话”的问题，本质上就是访问环境不满足安全上下文。

解决方案是：

- 使用正式域名。
- 通过 HTTPS 访问前端。
- API 和 Socket.IO 都走同域 HTTPS/WSS。
- App WebView 也配置为 HTTPS 域名。

这也是云部署中很典型的问题：代码逻辑没错，但协议和访问入口不正确，功能仍然无法使用。

## NAT、STUN 与 TURN

WebRTC 在局域网内通常容易成功，但跨网络、移动网络、校园网、运营商 NAT 时，点对点连接可能失败。这时需要：

- STUN：帮助双方发现公网映射地址。
- TURN：当 P2P 失败时中继音频流。

Echo 当前已经抽象了 RTC 配置，后续可以接入云厂商的 TURN 服务，或者自建 coturn。对于生产级 IM，TURN 几乎是必需项，因为不能假设所有用户网络都适合 P2P。

## 通话记录与用户体验

用户打完电话后，希望聊天记录里像微信一样出现“语音通话 0:34”。这不是 WebRTC 自动提供的，需要应用层记录：

- 通话接通时记录开始时间。
- 挂断时计算持续时间。
- 通过消息系统写入一条 `call` 类型消息。
- 会话列表把它显示为通话记录。

这体现了 IM 系统的一个设计原则：实时事件最终要回归消息模型，才能被历史记录、搜索、会话列表复用。

## 总结

Echo 的实时通信实践说明，云计算不是只会买服务器和开端口。真正上线一个通信应用，需要理解 HTTPS、WebSocket、WebRTC、NAT、STUN/TURN、反向代理和移动端权限。IM 项目是一个很适合云计算学生练综合能力的场景。
