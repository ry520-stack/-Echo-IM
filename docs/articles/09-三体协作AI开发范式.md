# 大二学生手搓 WebRTC：我是如何用「三体协作」AI 开发模式，一个人干完整个全栈社交 App 的

> 一个人，三个 AI，一台阿里云 ECS，从零到一跑通了实时通讯 + WebRTC 音视频通话。
> 这不是 Demo，是真正跑在生产环境里的全栈项目。

## 前言

我是一名大二云计算专业的学生。过去三个月，我独立开发了一个名为 **Echo** 的全栈即时通讯 App，技术栈是 React + Node.js + Socket.io + PostgreSQL + Docker，部署在阿里云 ECS 上。

这个项目包含的功能远不止聊天：
- 实时消息收发（WebSocket）
- 语音消息录制与播放（MediaRecorder API）
- **WebRTC 点对点语音通话**（零第三方服务，纯手搓）
- 星轨动态系统（朋友圈）+ 引力圈（收藏好友）
- 星域权限系统（可见/不可见分组，类似微信朋友圈分组）
- 自定义聊天背景 + 云端同步
- 回声胶囊（定时消息）
- 暗黑模式 + 毛玻璃 UI + CSS 硬件加速动画

但我今天不聊技术细节，我想聊一个更有趣的话题：**我是怎么用一种「三体协作」的 AI 开发模式，把这些功能全部落地的。**

## 技术高光：我们到底做了什么

### 1. Socket.io 信令服务器

我们的 Socket.io 服务不只是用来收发消息的。它同时扮演了 **WebRTC 信令服务器** 的角色，处理呼叫邀请、SDP 媒体协商、ICE 候选者交换等全部信令流程。

```typescript
// 后端信令转发 —— 极简但致命
socket.on('call:request', (data) => {
  socket.to(`user:${data.receiverId}`).emit('call:invite', {
    senderId: userId, callerName: data.callerName, callerAvatar: data.callerAvatar,
  });
});

socket.on('webrtc:signal', (data) => {
  socket.to(`user:${data.targetId}`).emit('webrtc:signal', {
    senderId: userId, signal: data.signal,
  });
});
```

### 2. 双引擎录音方案

语音消息同时支持 Web 浏览器和未来的 5+ App。核心是统一使用 `MediaRecorder` 标准 API，录音完成后上传到服务器返回 URL，再通过 Socket 发送 `type: 'voice'` 消息。

```typescript
const stream = await navigator.mediaDevices.getUserMedia({ audio: true });
const mediaRecorder = new MediaRecorder(stream);
mediaRecorder.start(200); // 每 200ms 采集一次
```

### 3. 零成本 WebRTC 语音通话

没有用任何第三方付费服务。利用 Google 公共 STUN 服务器做 NAT 穿透，通过 Socket.io 交换信令，在两台手机之间建立 P2P 音频直连。

```typescript
const rtcConfig: RTCConfiguration = {
  iceServers: [{ urls: 'stun:stun.l.google.com:19302' }],
};
const pc = new RTCPeerConnection(rtcConfig);
```

### 4. 星域权限系统

类似微信朋友圈的分组可见功能。数据库层面引入了 `FriendGroup`（分组）、`FriendGroupMember`（成员映射）、`MomentPermission`（权限判定表）三张表，通过 Prisma 的组合条件实现四条规则的精确过滤：

- 自己的动态永远可见
- `PUBLIC` — 所有好友可见
- `VISIBLE_TO` — 白名单分组可见
- `INVISIBLE_TO` — 黑名单分组不可见

## 核心亮点：三体协作开发模式

这是今天最想分享的部分。

在开发 Echo 的过程中，我摸索出了一种极其高效的 **「人类 + Gemini + Claude」三方协作模式**。这个模式的核心思想是：**每个角色只做自己最擅长的事。**

### 角色分工

| 角色 | 身份 | 职责 |
|------|------|------|
| **我（人类）** | 产品经理 + 主架构师 | 需求定义、流程把控、服务器部署运维、最终验证 |
| **Gemini** | 技术顾问 + Code Reviewer | 顶层架构方案、排雷防错、提供无 Bug 的高阶闭环代码 |
| **Claude** | 首席执行官 + 全栈研发 | 读取项目上下文、精准注入代码、跑通构建和部署 |

### 协作流程（以 WebRTC 通话功能为例）

**第一步：我发起需求**

> "我要给 Echo 加上语音通话功能。"

**第二步：Claude 收集上下文**

Claude 自动扫描项目代码，把所有相关组件（`ChatWindow.tsx`、`socket.service.ts`、`SocketContext.tsx` 等）复制到一个文件夹里。

```
在 D:\DevelopAPP\webrtc-debug\，4 个文件：

useWebRTC.ts        — 空文件（通话引擎 Hook，待 Gemini 注入）
CallScreen.tsx      — 空文件（呼叫/接听/挂断 UI，待 Gemini 注入）
ChatWindow.tsx      — 聊天窗口（需要加电话图标触发拨号）
socket.service.ts   — Socket.io 服务（需要加 call:request/call:accept/webrtc:ice 事件）
```

**第三步：我把文件夹发给 Gemini**

Gemini 拿到完整的项目上下文后，一次性输出了四个文件的完整代码：
- `useWebRTC.ts` — 完整的 WebRTC 通话引擎 Hook
- `CallScreen.tsx` — 高颜值通话全屏蒙层
- `socket.service.ts` 的信令事件补丁
- `ChatWindow.tsx` 的集成方案

**第四步：我把方案传达给 Claude**

Claude 理解 Gemini 的方案后，精准地将代码注入到项目中，处理好所有导入关系、类型检查、状态管理，然后自动构建、打包、部署到 ECS。

**第五步：我在手机浏览器上验证**

刷新页面，点电话图标，两台设备之间直接建立了语音通话。整个过程从需求到上线，不到 20 分钟。

### 为什么这个模式这么高效？

**1. 每个 AI 只做自己擅长的事**

Gemini 擅长**顶层设计和 Code Review**。它能一眼看出 `e.currentTarget` 在 `setTimeout` 中会失效、群聊消息会导致 `Duplicate Key`、`setInterval` 会产生雪崩效应这些隐蔽的 bug。这些坑如果靠我自己踩，可能要花半天时间调试。

Claude 擅长**理解大型代码库和精准执行**。它能读取项目里 50+ 个文件的完整上下文，理解组件之间的依赖关系，然后把 Gemini 给的方案无缝注入到正确的位置。

**2. 人类只需要做决策**

我不需要自己写代码，也不需要自己 debug。我只需要：
- 定义需求："我要语音通话"
- 传递信息："Gemini 说这样改"
- 验证结果："手机上能打电话了"

**3. 上下文隔离带来的质量保障**

Gemini 看不到我的完整项目（太大了），但它能看到我精选的、最相关的代码。这种「精选上下文」的方式，反而让它能更专注地分析问题，不会被无关代码干扰。

Claude 能看到完整项目，但它不需要从零设计方案。它只需要执行 Gemini 验证过的方案，减少了「自己瞎猜修 bug」的概率。

## 总结与感悟

### 「人类出脑子，AI 出算力」

这句话在 2025 年之前可能还是一句口号，但现在它已经是我的日常工作方式了。

作为独立开发者，你不需要精通所有技术栈。你需要的是：
1. **清晰的产品思维** — 知道自己要做什么
2. **基本的架构能力** — 知道技术栈怎么选
3. **懂得向 AI 提问** — 把问题描述清楚，把上下文给到位

### 对云原生架构的感悟

这个项目让我真正理解了「云原生」不是什么高大上的概念。它就是：
- Docker 容器化部署（一个 `docker-compose up` 搞定前后端 + 数据库）
- WebSocket 实时通信（Socket.io 真香）
- RESTful API 设计（Express 四件套：Controller / Service / Route / Middleware）
- 数据库建模（Prisma + PostgreSQL 的关系型设计）

当你把这些东西真正跑通一次，你就会发现：**所谓的「全栈工程师」，并没有想象中那么遥不可及。**

### 给同龄人的建议

如果你也是计算机相关专业的学生，我强烈建议你找一个完整的项目从头做到尾。不要只写 Todo List，不要只做课程设计。

找一个你真正想做的东西（对我来说是 IM），然后用 AI 辅助你把它做出来。在这个过程中，你会学到的不只是编程，还有：
- 系统设计思维
- 前后端联调经验
- 服务器运维能力
- 怎么和 AI 高效协作

**这些，才是你在校招面试中真正能拿出手的东西。**
